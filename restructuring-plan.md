# Mehmiro Documentation Restructuring Plan

## 1. Executive Summary
The goal is to transform the current developer-centric documentation into a multi-audience platform serving **End Users** (Teachers/Students), **Developers** (Contributors/Team), and **Researchers** (Academic/Pedagogical).

## 2. Proposed Site Architecture

### 2.1 Top-Level Navigation (Navbar)
The top navigation bar will be reorganized to clearly separate audiences:

| Label | Target Audience | Destination |
|-------|----------------|-------------|
| **User Guide** | Teachers, Students | `/docs/user/intro` |
| **Developer Hub** | Engineers, Contributors | `/docs/developer/intro` |
| **Research** | Academics, Pedagogues | `/docs/research/intro` |
| **Blog** | All | `/blog` |

### 2.2 Directory Structure
We will restructure the `docs/` directory to physically separate content by audience.

```
docs/
├── user/                  # [NEW] End-user documentation
│   ├── intro.md
│   ├── getting-started/
│   ├── tutorials/
│   └── guides/
├── developer/             # [MOVED] Existing technical docs
│   ├── intro.md
│   ├── architecture/
│   ├── standards/
│   ├── processes/
│   └── features/          # Technical specs (BDD, Implementation)
└── research/              # [NEW] Academic & Pedagogical content
    ├── intro.md
    ├── papers/
    └── methodology/
```

## 3. Content Organization Strategy

### 3.1 User Guide (New)
Focus: Practical usage, workflows, and "how-to".
- **Getting Started**: Account setup, first steps for Teachers vs Students.
- **Feature Guides**: Non-technical explanations of features (e.g., "How to interpret Assessment Insights" vs the technical "Assessment Insights Architecture").
- **Tutorials**: Step-by-step workflows (e.g., "Creating your first class").
- **FAQ**: Common questions.

### 3.2 Developer Hub (Refactored)
Focus: Implementation, architecture, and contribution.
- **Migration**: Move existing folders (`architecture`, `standards`, `processes`, `features`) into `docs/developer/`.
- **Foundations**: Split `foundations/`.
    - *Keep in Dev*: `persona.md`, `core-guidelines.md`.
    - *Move to General/Shared*: `project-overview.md`, `glossary.md` (or create audience-specific glossaries).
- **Features**: Keep the existing technical specs (BDD, Architecture) here.

### 3.3 Research Section (New)
Focus: Theoretical backing and academic validity.
- **Methodology**: Explaining the "Formative Feedback" model.
- **Papers**: Published research or whitepapers.
- **Case Studies**: Real-world application results.

## 4. Implementation Steps

### Phase 1: Structural Preparation
1.  Create new top-level directories: `docs/user`, `docs/developer`, `docs/research`.
2.  Move existing technical directories into `docs/developer`.
3.  Update `docusaurus.config.ts` and `sidebars.ts` to support multiple sidebars.

### Phase 2: Content Migration & Creation
1.  **Developer Hub**: Audit links in moved files (fix relative paths).
2.  **User Guide**: Create `intro.md` and skeleton structure for "Getting Started".
3.  **Research**: Create `intro.md` and placeholder for "Methodology".

### Phase 3: Navigation & UX
1.  Configure `sidebars.ts` to have distinct sidebars for each section:
    - `userSidebar`
    - `devSidebar`
    - `researchSidebar`
2.  Update Navbar in `docusaurus.config.ts`.
3.  Update Landing Page (`src/pages/index.tsx`) to direct users to their relevant section.
